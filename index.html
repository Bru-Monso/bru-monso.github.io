Proyecto 01

Integración de aplicaciones empresariales mediante Single Sign-On (SAML)

Contexto

A medida que Microsoft Entra ID fue adquiriendo un papel central dentro de la infraestructura, surgió la necesidad de integrar distintas aplicaciones empresariales mediante Single Sign-On.

El objetivo era centralizar la autenticación, simplificar la gestión de usuarios y aprovechar políticas de seguridad ya existentes como MFA, Conditional Access o la gestión basada en grupos.

Durante este proceso participé en varias integraciones utilizando SAML, colaborando inicialmente en Statista y realizando posteriormente de forma autónoma la integración de Sigma, AXES y la preparación de AWS IAM Identity Center.

Qué hice

Cada aplicación requería una configuración diferente, así que antes de configurarla era necesario entender cómo esperaba autenticarse cada proveedor y qué información necesitaba recibir desde Microsoft Entra ID.

  • Configuración de Enterprise Applications en Microsoft Entra ID.
  • Intercambio de metadata XML y certificados con proveedores.
  • Configuración de autenticación SAML.
  • Definición de claims, atributos de usuario y NameID.
  • Configuración de grupos dinámicos para automatizar la asignación de acceso.
  • Validación funcional junto con proveedores y usuarios finales.
  • Puesta en producción de las integraciones.

En Sigma, además, diseñé grupos dinámicos para separar automáticamente profesores y alumnos, ya que ambos colectivos acceden a aplicaciones diferentes utilizando el mismo proveedor de identidad.

La integración con AWS IAM Identity Center quedó preparada desde Microsoft Entra ID, quedando pendiente únicamente la configuración final dentro de AWS Organizations por limitaciones de la cuenta actual.

Lo más interesante

La parte más compleja no fue la configuración técnica de SAML, sino entender cómo funciona realmente el proceso de autenticación entre Microsoft Entra ID y cada aplicación.

Cada proveedor interpreta de forma diferente aspectos como los atributos, los claims, el NameID o los certificados. Eso obligó a entender la lógica de cada integración antes de configurarla bien.

Ese cambio de mentalidad fue importante: dejar de seguir documentación paso a paso y empezar a entender cómo diseñar este tipo de integraciones.

Qué aprendí

Este proyecto me permitió comprender mejor el funcionamiento de SAML y el papel de Microsoft Entra ID como proveedor de identidad.

También entendí la importancia de los atributos sincronizados desde Active Directory, los grupos dinámicos y la relación entre autenticación, autorización y gobierno de identidades.

Una buena integración no termina cuando un usuario inicia sesión. Tiene que facilitar la administración futura y permitir aplicar de forma centralizada medidas de seguridad como MFA, Conditional Access o, próximamente, Privileged Identity Management.

Resultado

Actualmente varias aplicaciones empresariales utilizan Microsoft Entra ID como proveedor de identidad, reduciendo la gestión manual de credenciales y facilitando una administración más centralizada del acceso a las aplicaciones.

Más allá de cada integración individual, este proyecto sentó las bases para seguir evolucionando el modelo de identidad de la organización y afrontar futuras implantaciones de forma más sencilla.

Área 02

Gobierno, seguridad y optimización cloud

Contexto

Trabajar en cloud no es solo desplegar servicios. También implica revisar seguridad, costes, accesos, backups y operación diaria para que el entorno sea mantenible.

Aquí agrupo proyectos relacionados con AWS, seguridad, optimización de costes y evolución de infraestructuras cloud.

Retos

  • Administrar infraestructura AWS existente sin tratarla como un entorno de laboratorio.
  • Analizar hallazgos de auditoría y convertirlos en un plan técnico accionable.
  • Mejorar el acceso a servidores reduciendo dependencia de SSH e IPs públicas.
  • Revisar IAM, usuarios, roles, MFA, access keys y principio de mínimo privilegio.
  • Detectar oportunidades de optimización de costes sin vender humo ni hablar solo de números.
  • Diseñar una base más clara de backups, versionado, lifecycle y recuperación.

Proyectos destacados

Próximos pasos

Evolucionar el entorno hacia un modelo más gobernado con AWS IAM Identity Center, Organizations, SCP, Security Hub, GuardDuty, Systems Manager y una estrategia más clara de automatización.

Área 03

Infraestructura híbrida

Contexto

En muchos entornos reales conviven Active Directory, Entra ID, servidores Windows y Linux, VMware, redes, firewalls, VPN y aplicaciones corporativas que tienen que seguir funcionando mientras la infraestructura evoluciona.

Este bloque reúne proyectos donde la infraestructura local y la nube conviven para ofrecer un entorno único, desde redes y servidores hasta servicios corporativos.

Retos

  • Entender dependencias entre sistemas antes de modificar nada.
  • Mantener servicios internos funcionando mientras se introducen mejoras.
  • Conectar identidad local, Entra ID, redes y servicios cloud de forma ordenada.
  • Resolver incidencias reales de infraestructura sin perder visión global del entorno.
  • Administrar sistemas corporativos que dependen tanto de software como de hardware.
  • Documentar y dejar las soluciones preparadas para que puedan mantenerse después.

Proyectos destacados

Próximos pasos

Seguir conectando la operación diaria de infraestructura con una visión más cloud, automatizada y documentada, sin perder de vista los sistemas que ya existen y que tienen que seguir funcionando.
